# 0.2.1 — 2026-07-13

## Changed

- **RxNorm wording reconciled** — server `instructions`, the `medcode_list_systems` description, and the Dockerfile's `org.opencontainers.image.description` LABEL said "Prescribable Content subset/snapshot"; now say "current normalized set", matching what `medcode_list_systems` actually reports and the RxNav-sourced scope shipped since v0.2.0. ([#12](https://github.com/cyanheads/medical-codes-mcp-server/issues/12))

## Removed

- **`medcode_search_codes`'s `no_match` error contract** — a zero-match FTS query always returned a successful empty result with a `ctx.enrich.notice`, never the declared error; removed the unreachable contract and its now-unused `JsonRpcErrorCode` import. No client-visible behavior change — a zero-match search was already, and remains, a successful empty result with recovery guidance. ([#15](https://github.com/cyanheads/medical-codes-mcp-server/issues/15))

## Dependencies

- **`@cyanheads/mcp-ts-core`** `^0.10.9` → `^0.10.14` — no server code changes; adopts five upstream patch releases.
- **Supply-chain hardening** — `bunfig.toml` gains a 3-day `minimumReleaseAge` hold on newly published package versions (excluding `@cyanheads/mcp-ts-core` via `minimumReleaseAgeExcludes` so framework releases adopt same-day) and the `@socketsecurity/bun-security-scanner` install scanner (`^1.1.2`, new devDependency; added to `devcheck.config.json`'s depcheck-ignore list since it's config-only, no direct import). `scripts/devcheck.ts`'s Outdated check now treats a release-age hold (bun's `*` marker) as nothing to adopt rather than a failure; `typescript` added to the Outdated allowlist.
