# 0.2.2 — 2026-07-13

## Added

- **Cursor pagination** — `medcode_search_codes`, `medcode_browse_hierarchy`, and `medcode_map_codes` (`children` and `name_to_rxcui` directions) accept an opaque `cursor` alongside `limit` and return a `nextCursor` when more results exist, disclosed on both `structuredContent` and `content[]`. ([#16](https://github.com/cyanheads/medical-codes-mcp-server/issues/16), [#17](https://github.com/cyanheads/medical-codes-mcp-server/issues/17), [#18](https://github.com/cyanheads/medical-codes-mcp-server/issues/18))
- **`childrenTruncated` on `medcode_get_code`** — with `includeHierarchy` set, a code with more immediate children than the cap now discloses `childrenTruncated: true`, pointing to `medcode_browse_hierarchy` / `medcode_map_codes` for the rest. ([#16](https://github.com/cyanheads/medical-codes-mcp-server/issues/16))
- **`CodeIndexService.fetchPage`** — shared `limit + 1` pagination primitive behind every capped query path (children, FTS search, drug-name crosswalk, browse), with deterministic `ORDER BY` tie-breakers (`bm25(codes_fts), c.system, c.code` for search; `length(code), code` for `name_to_rxcui`) so offset-based pages never skip or repeat rows.

## Fixed

- **`medcode_browse_hierarchy`** child results were silently capped at `MEDCODE_MAX_RESULTS` regardless of the requested `limit` — the node path never forwarded it to `childrenOf` — and `shown`/`cap`/`truncated` were computed against the input `limit` rather than the actual result. Both corrected. ([#16](https://github.com/cyanheads/medical-codes-mcp-server/issues/16))
- **`medcode_search_codes`'s `truncated`** used a `rows.length >= cap` heuristic that false-positived when the corpus held exactly `cap` matches — now exact via the `limit + 1` primitive. ([#17](https://github.com/cyanheads/medical-codes-mcp-server/issues/17))
